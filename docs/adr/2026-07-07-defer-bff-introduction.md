# BFF の導入を M8 相当の要件が生じるまで先送りする

Date: 2026-07-07

## Status

Accepted

## Context

design-doc 9.3 は BFF を「MVP では optional、本格運用では推奨」としており、導入時期は Open Question だった。
現状はブラウザから e-Gov 法令 API を直接取得しており、CORS を含めて動作している。
BFF が買えるものは、エッジキャッシュによる高速化、e-Gov 仕様変更からの絶縁、略称辞書のアプリリリース非依存な更新、全法令検索インデックス・AI キー秘匿・Web Push・同期 API の置き場所である。
一方で導入すれば、個人開発のプロダクトに常時運用するコンポーネントが 1 つ増える。

## Decision

BFF は当面導入せず、e-Gov API の直接取得を継続する。
次のいずれかが発生した時点で導入を再検討する。

- e-Gov API の CORS 方針や仕様変更で直接取得が壊れた。
- 全法令横断検索（M8）に着手する。
- AI 機能で API キーの秘匿が必要になった。
- Web Push またはときどき六法との同期 API に着手する。
- e-Gov の応答遅延が体感の問題として常態化した。

あわせて、先送りを安全にするための保険を維持する。

- 取得層は fetcher 注入と base URL の一点管理を保ち、プロキシへの切替を設定変更で済む状態にしておく（現行の LawRepository の構造を維持する）。
- e-Gov 応答のスキーマガード（構造の検証）をクライアント境界に置き、仕様変更を早期に検知する。
- 遅延対策は Service Worker のキャッシュ戦略（design-doc 9.5）で行う。
- 略称辞書はアプリ同梱とし、更新はリリースで行う（Resolver 仕様 2026-07-07 で決定済み）。

## Consequences

- 実装と運用の負担を増やさずに M4〜M7 を進められる。
- e-Gov の仕様変更にはリリースでの即応が必要になる（BFF による吸収層がない）。
- 略称辞書の更新頻度はアプリのリリース頻度に律速される。
- 再検討トリガーを明文化したため、「なんとなく先送りが続く」ことを防げる。導入時は本 ADR を Superseded にして新しい ADR を書く。

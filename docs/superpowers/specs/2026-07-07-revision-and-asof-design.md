# 改正・基準日の参照設計

Status: Approved (設計検討セッション 2026-07-07)
Last updated: 2026-07-07

関連ドキュメント:

- [Design Doc](../../design-doc.md) の 5.1 章（基準日で条文を確認したい）、7.4 章、16 章、19 章（storage eviction とは別のリスクとして「改正による参照の陳腐化」を扱う）。
- [ADR 2026-07-05: e-Gov Law ID を canonical URL に使う](../../adr/2026-07-05-egov-law-id-canonical-url.md)。

## 1. 決定事項の要約

- 保存物（ブックマーク、メモ、復習カード）の条文参照は **二重アンカー + 指紋検証**で持つ。
- 基準日（学習年度）は**グローバル設定**とし、ビューワー・検索・復習の根拠条文表示すべてに適用する。未設定時は「今日」として扱う。

## 2. 参照アンカー

保存物が共通で使う参照型を次のとおり定める。

```ts
interface LawReferenceTarget {
  lawId: string;
  path: string; // 例: "part:1/chapter:1/article:709"
  revisionId: string; // 作成時点の版。常に開ける退避先
  fingerprint: string; // 条文指紋（3 章）
  createdAt: ISODateString;
}
```

path は既存の LawNode.path と同じ形式を使う。
revisionId は「作成時にユーザーが見ていた条文」を将来も再現するための退避先であり、既定の解決先ではない。

## 3. 条文指紋

- 対象: 参照先の条ノード（Article）の plainText 全文。
- 計算: NFKC 正規化 → 空白除去 → SHA-256 → 16 進表現の先頭 16 文字。
- 目的は改変検知であり、衝突耐性は要求しない。
- 全文を対象にするのは、条の後半だけの改正を見逃さないためである。句読点程度の改正でも不一致になるが、それは「改正の可能性」を知らせたい場面そのものなので許容する。

## 4. 解決規則

参照を開くときの手順を次のとおり定める。

1. 基準日で版を選ぶ: 「施行日 ≤ 基準日」を満たす最新版。
2. その版の中で path を解決する。
3. 解決した条文から指紋を再計算し、保存済み指紋と比較する。
4. 一致すればそのまま表示する（改正があっても当該条文が無傷なら通知しない）。
5. 不一致なら「改正の可能性」バッジを表示する。バッジから、作成時の版（revisionId）と現在の解決先を見比べる画面へ遷移できる。

見比べ画面でのユーザーの選択肢は 2 つに限定する。

- **新しい条文に付け替える**: fingerprint と revisionId を現在の解決先で更新する。
- **この版のまま固定する**: 以後この参照は revisionId 固定で開き、バッジを常設する。

path 解決自体が失敗した場合（条の削除・繰り下げ）は、指紋不一致と同じ導線に「現在の版に該当する条が見つかりません」という文言で合流させる。

## 5. 基準日

- 定義: グローバル設定「学習年度の基準日」。設定画面（学習グループ）から変更する。
- 適用範囲: 法令本文の表示、検索対象の版、復習カードの根拠条文表示のすべて。
- 未設定時は「今日」として扱う（常に現行法）。
- ビューワーのツールバーとフッターには解決に使った基準日を表示する（UI 設計で定義済み）。

## 6. オフライン保存との整合

- オフライン保存されるのは「保存操作の時点で基準日により解決された版」である。
- 基準日を変更しても保存済みデータは自動で差し替えない。設定画面のオフライン管理に法令ごとの「更新あり」を表示し、ユーザー操作で再同期する（UI 設計 8 章の方針と一貫）。
- 意図的に旧版のまま学習する使い方（改正前の条文で受験する）を壊さないことを優先する。

## 7. URL 契約

- `/laws/:lawId/articles/:article` は**基準日依存 URL**: 開いた人の基準日設定で版を解決する。
- `/laws/:lawId/:revisionId/articles/:article` は**版固定 URL**: 共有と、ときどき六法（Android）連携にはこちらを使う。
- 保存物から共有リンクを発行するときは、そのとき表示している版の版固定 URL を出す（受け取り側の基準日設定に左右されない）。

## 8. 前提と縮退

本設計は「e-Gov API v2 が法令ごとの改正版一覧と版指定取得を提供する」ことを前提にする。
実装着手時に API 仕様を検証するタスクを最初に置く。
前提が満たされない場合は、指紋検証を「取得のたびに前回スナップショットと比較する」方式へ縮退し、見比べ画面の旧版側はローカル保存済みスナップショットに限定する。

## 9. データ移行

- 保存済み法令（SavedLaw）は既に revisionId を持つため変更しない。
- ブックマーク・メモ・復習カードは未実装のため、本設計の LawReferenceTarget を初版スキーマとして導入する（破壊的移行なし）。
- IndexedDB のスキーマ変更は [ADR 2026-07-06: IndexedDB storage version 1](../../adr/2026-07-06-indexeddb-storage-version-1.md) の後続バージョンとして扱う。

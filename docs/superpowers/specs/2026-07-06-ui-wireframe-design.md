# すらすら六法 UI ワイヤーフレーム設計

Status: Approved (デザイン検討セッション 2026-07-06)
Last updated: 2026-07-06

関連ドキュメント:

- [Design Doc](../../design-doc.md) の 15 章（UI/UX Draft）を具体化するものである。
- HTML モックアップ: [docs/mockups/](../../mockups/)（単体で開ける最終版のみ。`index.html` が入口）

## 1. 決定事項の要約

デスクトップの法令ビューワーを 3 カラムで組む前提のもと、次の 5 点を対話で決定した。

- **タイポグラフィ**：本文と法令の文脈テキスト（法令名、条見出し、目次）は明朝体、操作部品（ボタン、タブ、ナビ）はゴシック体とする。
- **3 カラムの役割**：グローバルナビをヘッダーへ移し、左カラムは目次、中央は本文、右カラムは選択条文に連動するコンテキストパネルとする（design-doc 15.2 準拠）。
- **Home の性格**：上部を検索ランチャー、下部を学習ダッシュボードとする。学習データが空のときはランチャーだけの画面に退化する。
- **アクセントカラー**：六法全書の表紙を想起させる深緑とする。
- **選択中条文の表示**：カードやバッジではなく、左余白に置くブラケット（範囲括弧）で示す。

## 2. デザイントークン

### 2.1 カラー

| 役割             | 値                                | 用途                                     |
| ---------------- | --------------------------------- | ---------------------------------------- |
| アクセント       | `#166534`                         | 選択状態、主要ボタン、リンク、ブラケット |
| アクセント淡     | `#dcfce7` / `#f0fdf4`             | 選択ハイライト、バッジ地、正解表示       |
| アクセント深     | `#14532d`                         | 淡地上のテキスト                         |
| 背景（本文）     | `#fcfbf8`                         | 生成り。読書領域の地                     |
| 背景（パネル）   | `#faf8f2`                         | 左右カラムの地                           |
| 背景（ヘッダー） | `#fffdf9`                         | ヘッダーとフッター                       |
| 罫線             | `#e7e0d3` / `#d6cdbc`             | 区切り、入力枠                           |
| 墨               | `#27272a`                         | 本文                                     |
| 淡墨             | `#57534e` / `#78716c` / `#a8a29e` | 条見出しのかっこ書き、補足、無効         |

ダークモードではアクセント各色の明度を上げた変種を使う方針とし、具体値は残課題とする（9 章）。

### 2.2 タイポグラフィ

- **明朝**（`Hiragino Mincho ProN` 系）：条文本文、法令名、条見出し、目次の条タイトル、検索結果の条文抜粋、Home のキャッチコピー。
- **ゴシック**（`Hiragino Sans` 系）：ボタン、タブ、ナビゲーション、設定項目、メタ情報（取得日時、確信度）。
- 本文の行間は 2.0 前後とする。法令は一文が長く、通常の Web 標準（1.5〜1.7）では行の追跡を誤りやすいためである。

使い分けの規則は「読むものは明朝、操作するものはゴシック」の一文に集約できる。
画面単位でも同じ規則を適用し、読み物でない設定画面では明朝の出番を法令名だけに絞る。

### 2.3 構造表示の原則

条や項の番号に装飾（バッジ、チップ）を付けず、原文の佇まいを保つ。

- 条見出しは「第1条（公務員の不法行為と賠償責任）」の形で明朝の見出しとして組む。かっこ書きは本文よりわずかに淡い墨色にする。
- 項番号は原文どおり「２　前項の場合において…」と全角数字で組む。
- 漢数字から算用数字への変換（design-doc 12 章）は表示レイヤーで行い、この佇まいを変えない。

アクセント色の出番は「選択状態、目次ハイライト、定義語リンク、アクションボタン」に限定する。
装飾を減らした結果、条の区切りは余白と選択ブラケットだけが担うため、条間の余白は実装時に実データで調整する。

### 2.4 選択ブラケット

選択中の条文は、本文の左余白に置く深緑のブラケット（上下端の短い横線を持つ縦線）で示す。
本文領域そのものには一切手を加えない。

同じブラケットを復習クイズの根拠条文表示にも使う。
クイズからビューワーへ遷移したとき同じ印が同じ条文に付くため、画面間で「いま注目している条文」の表現が揃う。

## 3. 法令ビューワー

モックアップ: `screen-viewer-v2.html`（選択表示は `selection-style.html` の C 案で上書き）

```text
┌──────────────────────────────────────────────────┐
│ ヘッダー: ロゴ | 検索(/) | 法令 撮る 復習 設定      │
├─────────┬──────────────────────────┬─────────────┤
│ 目次     │ 本文ツールバー             │ コンテキスト │
│ 法令名   │ (表示切替/文字サイズ/       │ 選択中: 第1条│
│ 法令番号  │  法令内検索/基準日)        │ [メモ|定義語 │
│ 保存状態  │                          │  |関連|復習] │
│ 条リスト  │ ⌐ 第1条（見出し）          │ メモカード   │
│         │ | 本文…                   │ +メモ追加    │
│         │ ⌐ 第2条（見出し）           │ 復習カード化 │
│         │   本文…                   │ ブックマーク │
├─────────┴──────────────────────────┴─────────────┤
│ フッター: 出典 e-Gov ・取得日時・加工表示の注記      │
└──────────────────────────────────────────────────┘
```

- 左カラムには法令名、法令番号、オフライン保存状態、目次、条番号ジャンプを置く。目次の選択行は左バーと淡緑で示す。
- 本文ツールバーに「読みやすい表示 / 原文」切替、文字サイズ、法令内検索、基準日と取得日時を常設する。法的正確性の要件（design-doc 8.4）を読書中も満たすためである。
- 条にポインタを合わせると、条見出しの右端に保存、カード化、コピー、permalink のアクションが現れる。
- 右カラムは選択中の条文に連動し、メモ、定義語、関連、復習のタブを持つ。下部に「復習カードを作る」「ブックマーク」を固定する。
- フッターに出典と「表示は読みやすい加工を含む」旨を常設する。

条の選択、保存、復習カード化、permalink はいずれも同じ粒度（データモデルの `LawNode` の `path`）で動く。
UI の単位とデータの単位が一致するため、実装の分岐が減る。

## 4. Home

モックアップ: `screen-home.html`

- 上 1/3 は検索ランチャーとする。キャッチコピー、検索ボックス、「撮って開く」「貼り付けて開く」を置く。
- 下 2/3 は学習ダッシュボードとする。「今日の復習」バナー、最近開いた条文、ブックマーク、オフライン保存状態を置く。
- 「今日の復習」だけは全幅と深緑の塗りで別格に扱う。このアプリで唯一、アプリ側からユーザーに行動を求める要素だからである。復習が 0 件の日はバナーごと消す。
- 学習データが空のとき（初回起動）はダッシュボードを描画せず、コピーを大きくした検索ランチャーと「よく読まれている法令」チップ（固定リスト）だけを表示する。

## 5. 検索とジャンプ

モックアップ: `screen-search.html`

検索は 2 層に分ける。

- **コマンドパレット**：全画面から `/` キーまたはヘッダー検索で開くオーバーレイ。条文参照（`国賠1` など）の解決とジャンプを担う。
- **検索結果ページ**：本文横断検索（`賠償` など）の比較検討を担う。

パレットでは Resolver（design-doc 11 章）の確信度を面積と色で表現する。
最有力候補は淡緑の地で大きく表示し、Enter で即座に開ける位置にフォーカスを置く。
低確信の候補は無地で小さく続ける。
design-doc 11.4 の「閾値以上なら直接ジャンプ」は、UI 上は「Enter のデフォルトフォーカス」として実装する。
直接遷移と違い、誤解決でも esc で戻れる。

検索結果ページには「保存済みから / すべての法令（オンライン）」の範囲トグルを置き、いまどの範囲を検索しているかを常に示す。
オフライン時は後者を無効化し、理由をツールチップで示す。

現行ルーティングの `/jump` 画面はパレットに統合して廃止する。

## 6. 撮る（OCR 取り込み）

モックアップ: `screen-scan-v2.html`（入口）、`screen-scan.html`（結果確認）

入口のアクションは 2 つに絞る。

- **撮る・画像を選ぶ**：`<input type="file" accept="image/*">` に委ね、カメラとライブラリの分岐は OS の選択シートに任せる。アプリ側では選ばせない。
- **クリップボードから貼り付け**：デスクトップでは画面全体をドロップゾーンにし、ファイル選択とペーストも同じゾーンで受ける。

テキスト手入力の入口は置かない。コマンドパレットが既にその役割を担っている。

プライバシー宣言「画像は端末内で処理され、保存・送信されません」は見出し直下の定位置に置く。
撮る対象が著作物（問題集）であるため、この安心感が利用率を左右する。

結果確認画面は左に撮影画像、右に検出した条文参照の候補カードを並べる。

- OCR 結果の全文テキストは表示しない（design-doc 13.2）。
- 確信度は数値で見せず、高は実線の緑枠、低は破線のグレー枠として、画像上のハイライトと候補カードを同じ文法で対応させる。
- 候補ごとに「開く」「復習に追加」「無視」を置き、出口に「まとめて復習リストへ」を固定する。問題集 1 ページに参照が複数あるのが常態だからである。

## 7. 復習

モックアップ: `screen-study.html`

復習ホームには「今日の復習」バナー（Home と同じ文法）、苦手な条文、カード内訳、科目別プリセットを置く。
習慣化の演出は「今週 2/5 日」の細いバーだけにとどめ、連続記録の煽りは入れない。

クイズ画面では回答後に必ず根拠条文を表示する（design-doc 14.3）。

- 根拠条文には選択ブラケットを付け、タップでビューワーの該当条文へ遷移する。
- 評価ボタンは「もう一度 / 難しい / 普通 / 簡単」の 4 択とし、それぞれの下に次回復習までの間隔（例: 2日、4日）を小さく併記する。間隔反復の仕組みを説明なしで体感させるためである。

## 8. 設定とオフライン管理

モックアップ: `screen-settings.html`

設定トップは OS の設定画面の文法（グループ + 行 + 右端に現在値）で組む。
グループは表示（文字サイズ、行間、テーマ、既定の表示）、学習（基準日、科目プリセット）、データ（オフライン管理、エクスポート/インポート、ときどき六法連携）とする。
フッターに「本アプリは学習補助であり、法的助言を提供するものではありません」を常設する。

オフライン管理画面には次を置く。

- ストレージ使用量バーと最終同期日時、全体再同期。
- 永続ストレージ（`navigator.storage.persist()`）の許可状態バナー。未許可なら「保護を有効にする」ボタンに変わる。design-doc 19 章のリスク「PWA storage eviction」への UI 側の回答である。
- 保存済み法令の一覧（サイズ、同期日、個別削除）。e-Gov 側に改正があれば行単位で「更新あり」を示す。

「更新あり」を自動適用しないのは、資格学習者が学習年度の基準日で条文を固定したい（改正前の条文で試験を受ける）場合があるためである（design-doc 5.1）。

## 9. 残課題（今回のスコープ外）

次は今回のセッションで扱っておらず、別途デザイン検討が必要である。

- モバイル各画面の詳細。ボトムナビの構成は現行 AppShell を踏襲する前提だが、ビューワーの目次とコンテキストパネルをボトムシートにするか否かは未決定。
- ダークモードの具体的なトークン値。
- 空状態以外のエッジケース（オフライン時のビューワー、e-Gov 取得失敗時のエラー表示）。
- ブックマーク一覧画面（`/bookmarks`）。Home のカードからの遷移先として必要になる。

## 10. 実装への影響

現行コードとの差分は次のとおりである。

- `AppShell.tsx` の左カラムからグローバルナビを撤去し、ヘッダーへ移す。左カラムは法令ビューワー専用の目次となるため、AppShell 共通ではなくビューワー画面の持ち物になる。
- 右カラム（学習パネル）を「選択条文連動のコンテキストパネル」として作り直す。
- `/jump` ルートを廃止し、コマンドパレットに統合する。
- Tailwind のテーマに 2 章のカラートークンとフォントファミリーを定義する。

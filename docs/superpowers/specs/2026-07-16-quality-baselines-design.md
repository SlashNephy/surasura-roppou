## 1. 目的

Issue #40 として、アクセシビリティ、性能、プライバシー、データ取扱いの最低基準と検証手順を定める。

基準は、実装者が機能設計、実装、レビュー、実画面確認で繰り返し参照できる形にする。

現在の実装が基準を満たすかも監査し、未達項目は測定値や再現手順を付けた後続 Issue へ分離する。

## 2. 成果物と完了境界

Issue #40 の成果物は次の三つとする。

1. `docs/quality-baseline.md` に最低基準、現在の監査結果、検証手順、実装チェックリストをまとめる。
2. `docs/design-doc.md` から品質基準文書へリンクし、Design Doc は製品方針、品質基準文書は具体的な判定方法という役割に分ける。
3. 現在未実装の表示設定とローカルデータ削除を後続 Issue にし、監査で別の未達項目が見つかった場合も証跡付き Issue にする。

Issue #40 では、文字サイズ、行間、テーマの設定 UI とローカルデータ削除機能を実装しない。

性能改善、アクセシビリティ修正、analytics 基盤の導入も対象外とする。

ただし、基準を決めるだけでは現状を確認したことにならないため、実レスポンス、実ビルド、実ブラウザを使った監査は Issue #40 に含める。

## 4. 現在の実装から確認できる事実

### 4.1 アクセシビリティ

`src/app/AppShell.tsx` は header、nav、main、footer のランドマークを持ち、ナビゲーションにはアクセシブルネームがある。

共通 UI と主要リンクには `focus-visible` の表示があるが、アプリ全体の Tab 順、ダイアログのフォーカス復帰、モバイル導線を通したキーボード監査はまだ記録されていない。

したがって、コード上の属性だけで適合と判断せず、実画面で主要フローを操作する。

### 4.2 表示設定

`src/index.css` にはライトテーマとダークテーマのトークンがある。

一方、`src/app/settings-page.tsx` の文字サイズ、行間、テーマは静的な表示であり、利用者が変更できない。

`next-themes` は通知 UI から参照されているが、アプリルートにはテーマプロバイダーが接続されていない。

このため、Design Doc の「調整可能にする」という要件は未達として後続 Issue にする。

### 4.3 法令本文の描画

`src/core/viewer/LawNodeList.tsx` は、受け取った全ノードを再帰的に React 要素へ変換する。

仮想化や表示範囲による遅延描画は実装されていないため、大規模法令での main thread 占有時間を実測する必要がある。

2026-07-16 の予備ビルドでは、初期 JavaScript が 686.76 kB、gzip 後 206.66 kB となり、Vite の 500 kB 警告が出た。

この値は正式な監査値ではなく、実装計画で同じ条件から再測定する。

### 4.4 OCR の読み込み

`src/core/ocr/recognizer.ts` は、初回の worker 生成時に `import("tesseract.js")` を実行する。

2026-07-16 の予備ビルドでは OCR 関連の動的 chunk が 17.23 kB、gzip 後 7.30 kB として生成された。

一方、同じビルドの Service Worker precache manifest には、この動的 chunk が含まれていた。

Worker、WASM、日本語モデルは precache から除外されているが、OCR engine のコードは Service Worker のインストール時に取得されるため、現在の実装は本設計の lazy load 基準を満たさない可能性がある。

実ブラウザの Network と Application の状態を使って、初期表示、スキャナー表示、同意、認識開始の各時点を確認する。

### 4.5 OCR 画像と OCR テキスト

取り込んだ OCR 画像は `Blob` と object URL としてメモリ上で扱い、差し替えまたはアンマウント時に object URL を解放する。

保存スキーマの `OcrSession` は `sourceText` と検出参照を保持するが、画像フィールドを持たない。

したがって、画像を保存しない方針と、OCR テキストを端末内へ保存する現在の挙動は両立する。

OCR セッションは現行の export version 2 に含まれない。

### 4.6 analytics とローカルデータ削除

現行のアプリコードには analytics の import、設定、イベント送信処理がない。

lockfile には Workbox などの推移的依存として analytics 関連 package が含まれるため、依存 package 名の不在を根拠にしない。

`src/core/storage/repository.ts` の `deleteSurasuraDatabase` はアプリ本体の IndexedDB を削除できるが、利用者向け UI はない。

この関数だけでは、名前空間付き `localStorage`、Service Worker の Cache Storage、OCR モデルのキャッシュを含む完全な初期化にはならない。

## 5. アクセシビリティ基準

### 5.1 適合目標

Web UI は [WCAG 2.2](https://www.w3.org/TR/WCAG22/) の Level AA を適合目標とする。

自動検査だけでは、論理的なフォーカス順や操作後の状態理解を判定できないため、主要フローはキーボードで通して確認する。

### 5.2 キーボード操作

ポインターで実行できる主要操作には、キーボードから同等の操作方法を用意する。

Tab と Shift+Tab の順序は視覚上の順序と作業の流れを壊さないものとし、正の `tabIndex` で順序を上書きしない。

フォーカス表示は常に視認でき、sticky header、mobile navigation、dialog、toast に完全に隠れないようにする。

dialog と sheet は Escape で閉じられ、閉じた後は起動元へフォーカスを戻す。

フォーカストラップは modal の表示中だけに限定し、ページ本体でキーボード利用者を閉じ込めない。

### 5.3 構造と状態通知

各画面は一意に理解できる `h1` と main ランドマークを持つ。

法令本文の編、章、節、条、項、号は、視覚上の字下げだけに依存せず、見出し、article、list、ラベルの組み合わせで構造を伝える。

アイコンだけのボタンには操作対象を含むアクセシブルネームを付ける。

入力エラーは対象入力と関連付け、処理成功は `role="status"`、失敗は `role="alert"` などで支援技術へ伝える。

色だけで選択中、失敗、警告、保存状態を区別しない。

### 5.4 画面拡大とタッチ

200% の画面拡大と 320 CSS px 幅をそれぞれ確認し、主要情報と操作を失わず横方向のページスクロールを発生させない。

法令の表など二次元配置が意味の理解に必要な内容は、その内容を囲む領域だけで横方向にスクロールできるようにし、ページ全体を横スクロールさせない。

単独のボタン、ナビゲーション、カード操作のタッチ領域は 44 x 44 CSS px 以上を推奨し、24 x 24 CSS px を下回らない。

本文中のインラインリンクは WCAG 2.2 の例外を適用できるが、隣接リンクとの識別とキーボードフォーカスを維持する。

hover だけで現れる操作を唯一の導線にしない。

## 6. 文字サイズ、行間、テーマの方針

文字サイズ、行間、テーマは表示レイヤーの利用者設定とし、法令原文、保存データ、コピー内容を変更しない。

後続 Issue は、少なくとも次の選択肢を提供する。

- 文字サイズ：標準、やや大きい、大きい
- 行間：標準、ゆったり、広い
- テーマ：システム、ライト、ダーク

数値は後続 Issue の設計と実画面比較で決めるが、ブラウザの文字拡大とページ拡大を妨げないことを先に満たす。

初期テーマはシステム設定へ追従し、利用者が明示的に選んだ値だけを名前空間付きローカル設定へ保存する。

表示設定を analytics へ送らない。

サーバー同期を将来追加する場合も、同期対象へ含めるかを利用者が選べるようにする。

## 7. 性能基準と測定方法

### 7.1 製品目標と監査値

実利用環境の目標は Core Web Vitals の good threshold とする。

- LCP：2.5 秒以下
- INP：200 ms 以下
- CLS：0.1 以下

判定は mobile と desktop を分け、実利用データの 75 パーセンタイルで行う。

Issue #40 のローカル測定には実利用データがないため、Core Web Vitals の適合を宣言しない。

ローカル測定は問題箇所と回帰比較の基準を作るための監査値として記録する。

### 7.2 アプリシェル

本番ビルドは、初期 JavaScript に OCR engine、OCR worker、WASM、日本語モデルを含めない。

Service Worker の precache にも OCR engine の動的 chunk、worker、WASM、日本語モデルを含めない。

Vite の既定 chunk size 警告を解消することを目標とし、警告が残る場合は chunk ごとの責務とサイズを記録した改善 Issue を作る。

PWA precache の総量と初期 JavaScript の minified/gzip サイズを監査結果へ記載する。

### 7.3 大規模法令本文

代表データには e-Gov API から取得した民法（lawId `129AC0000000089`）の実レスポンスを使う。

測定時に取得した revisionId、レスポンスサイズ、正規化後のノード数を記録し、別の小さな手書き fixture で代用しない。

production build と headed Chromium を使い、cold cache と warm cache を各 3 回測定して中央値を記録する。

測定では次を分けて記録する。

1. e-Gov API の応答時間
2. レスポンスの変換と正規化に要した時間
3. React が法令本文を表示するまでの時間
4. 表示時の DOM 要素数
5. 50 ms を超える Long Task の件数と最大時間
6. 表示後の目次ジャンプ、表示モード切替、スクロール操作の応答

50 ms を超える処理は Long Task として記録する。

ローカル監査では、次のいずれかに該当した場合を未達として性能改善 Issue を作る。

- warm cache でレスポンス受信完了から最初の条文表示までの中央値が 1,000 ms を超える。
- 目次ジャンプまたは表示モード切替から次の描画までの中央値が 200 ms を超える。
- 法令データの変換または初回描画によって 50 ms を超える Long Task が発生する。
- 大規模本文の表示中に操作不能、描画停止、メモリ不足、未処理例外が発生する。

1,000 ms はネットワーク時間を除いた法令表示のローカル guardrail とし、実利用環境の Core Web Vitals と同一の指標として扱わない。

一回の Long Task でも未達として記録するが、その証跡だけで仮想化を採用すると決めない。

改善 Issue は、時間、発生箇所、操作への影響を示したうえで解決方法を設計する。

### 7.4 測定環境

品質基準文書へ、OS、CPU、メモリ、ブラウザとバージョン、viewport、build commit、測定日時を記載する。

端末差を隠すための補正は行わず、同じ環境で比較できる条件を残す。

ネットワーク時間と描画時間を混ぜない。

## 8. OCR の lazy load 基準

OCR の実装は、利用者がスキャナー画面を開いただけでは OCR engine、Worker、WASM、日本語モデルを読み込まない。

利用者がモデル取得へ同意し、認識を開始した時点で初めて読み込む。

次の四段階で Network と Cache Storage または IndexedDB を確認する。

1. アプリの初期表示
2. スキャナー画面の表示
3. OCR モデル取得の同意表示
4. 同意後の認識開始

段階 1 から 3 では、OCR engine の動的 chunk、worker、WASM、日本語モデルの新規取得を禁止する。

段階 4 では、自オリジンから必要な資産を取得し、画像 Blob を request body や query string として送らない。

認識のキャンセルと画面離脱時には worker を解放する。

キャンセル後の再試行に選択画像を使えるよう、object URL は画像の差し替え、破棄、画面離脱時に解放する。

ビルド出力では、`tesseract.js` が初期 chunk と別の chunk になり、その動的 chunk が Service Worker の precache manifest に含まれないことを確認する。

## 9. プライバシーとセキュリティのデータ境界

### 9.1 保存と送信

データ分類ごとの基準は次のとおりとする。

| データ                               | 端末内保存           | export                    | analytics | 外部送信                                |
| ------------------------------------ | -------------------- | ------------------------- | --------- | --------------------------------------- |
| OCR 画像                             | 禁止                 | 禁止                      | 禁止      | サーバー OCR へ明示同意した場合だけ許可 |
| OCR テキストと検出参照               | `ocrSessions` に許可 | 現行 version 2 では対象外 | 禁止      | 禁止                                    |
| ブックマークとメモ                   | 許可                 | 利用者操作時だけ許可      | 禁止      | 同期機能への明示同意がある場合だけ許可  |
| 学習カード、回答ログ、学習セッション | 許可                 | 利用者操作時だけ許可      | 禁止      | 同期機能への明示同意がある場合だけ許可  |
| 表示設定と学習年度                   | 許可                 | 現行 version 2 では対象外 | 禁止      | 禁止                                    |

「端末内処理」は、JavaScript から第三者 API へ送らないことだけを意味しない。

画像、OCR テキスト、メモ、学習内容を URL、console、例外メッセージ、analytics event、エラー報告 payload に含めないことまでを含む。

### 9.2 analytics の導入条件

現時点では analytics を導入しない。

将来導入する場合は、送信可能な event 名とフィールドを allowlist で定義し、自由形式の payload を受け付けない別設計を先に承認する。

法令 ID、条文パス、検索語、メモ、OCR テキスト、学習履歴、ファイル名、画像由来の metadata は allowlist に含めない。

### 9.3 外部データの表示

e-Gov API、import file、OCR text は信頼済み HTML として扱わない。

React の text rendering または明示的に検証した構造へ変換し、`dangerouslySetInnerHTML` をデータ表示の近道として使わない。

外部リンクは HTTPS を使い、新しいタブで開く場合は opener への参照を残さない。

依存関係の脆弱性対応は Renovate と通常のレビュー手順で管理し、Issue #40 では依存更新を行わない。

## 10. ローカルデータ削除の方針

ローカルデータ削除は後続 Issue で実装する。

後続 Issue は、削除前に export 導線を示し、削除対象を利用者へ列挙し、取り消せない操作であることを確認してから実行する。

現行の export version 2 は OCR セッション、表示設定、OCR モデルのキャッシュを含まないため、全削除からの完全な復元手段にはならない。

後続 Issue は、export で復元できる七分類と復元できないデータを削除確認画面で分けて示す。

復元できないデータを削除対象へ含める場合は、その内容を明示した確認を export 操作とは別に求める。

削除対象は実装時に次の領域を棚卸しする。

- アプリの IndexedDB
- `surasura:` 名前空間の `localStorage`
- アプリが所有する Cache Storage
- Tesseract.js が作成する OCR モデルのキャッシュ
- 将来追加される表示設定と、現行のローカル検索データ

アプリが所有していると確認できない localStorage key、database、cache を削除しない。

`localStorage.clear()` のように origin 全体を無条件で消す処理は使わない。

削除前に repository の接続を閉じ、削除後は再読み込みして保存データ、設定、OCR cache が残っていないことを公開動作として検証する。

## 11. 監査手順

### 11.1 静的な棚卸し

`rg` と依存関係一覧を使い、analytics 送信、画像永続化、`dangerouslySetInnerHTML`、テーマ配線、削除対象の storage を棚卸しする。

文字列が見つからないことだけで動作を保証せず、実ブラウザの Network、Application、アクセシビリティツリーと組み合わせる。

### 11.2 実レスポンス

民法の実レスポンスを最低 1 件取得し、HTTP status、取得 URL、revisionId、payload size、正規化後 node count を記録する。

取得に失敗した場合は認証、CORS、API status、ネットワークを切り分けて再試行する。

実レスポンスを取得できないまま、大規模本文の性能を確認済みとしない。

### 11.3 実ブラウザ

システムにインストールされた `playwright-cli` を `open --headed` で起動し、同じセッションを再利用する。

デスクトップとモバイルで主要 route を開き、Tab、Shift+Tab、Enter、Space、Escape、矢印キーを使って主要操作を確認する。

OCR は Network の記録を消去してから四段階の読み込み境界を確認する。

大規模法令は同一 build で cold cache と warm cache を測定し、画面を開くだけでなく目次ジャンプ、表示モード切替、スクロールを操作する。

### 11.4 監査結果の表現

各項目を次のいずれかで記録する。

- **適合**：基準を満たす証跡がある。
- **未達**：基準を満たさず、後続 Issue がある。
- **未検証**：必要な実レスポンス、環境、操作を確認できていない。
- **対象外**：理由を文書に記載している。

「未達」を「適合」と書き換えて Issue #40 を閉じない。

修正を別 Issue へ分離している場合は、Issue URL と未達の証跡を監査結果へ残す。

## 12. 後続 Issue

Issue #40 の完了までに、次の二つを作成する。

1. 文字サイズ、行間、テーマ設定を実装する Issue
2. ローカルデータを安全に全削除する Issue

性能、キーボード操作、focus、privacy boundary に未達が見つかった場合は、領域ごとに再現手順と測定値を付けた Issue を追加する。

Service Worker が OCR engine の動的 chunk を同意前に precache する現状が実ブラウザでも再現した場合は、OCR lazy load の改善 Issue を追加する。

## 13. テストと完了確認

Issue #40 は production code を変更しないため、新しい自動テストを追加しない。

docs や設定ファイルを読み、特定の文字列があるかだけを確認するテストも追加しない。

コミット直前に次の標準チェックを実行する。

```bash
pnpm run typecheck
pnpm run lint

```

PR 前に `pnpm run review:antigravity` を実行する。

監査に使う `pnpm build`、実レスポンス取得、headed browser 操作は、標準チェックとは別の証跡として PR 本文に記載する。

## 14. エラー時の扱い

公的基準の参照先が取得できない場合は、取得済みの文面を断定的に更新せず、再取得して現行版を確認する。

性能値のばらつきが大きい場合は、最良値だけを採用せず、各試行値と中央値を残す。

実ブラウザで操作不能や console error を見つけた場合は、文書だけを適合へ変更せず、再現手順付きの未達として記録する。

Antigravity CLI が利用できない場合は skip の出力を PR と最終報告へ記録する。

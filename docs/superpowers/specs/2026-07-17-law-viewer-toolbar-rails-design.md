# 条文ビューワー ツールバー整理（2レール分業）設計

## 背景と目的

条文ビューワー（`src/app/law-viewer-page.tsx`）は、中央本文の上部に大きな操作カード（表示モード切替・条番号ジャンプ・目次トグル・オフライン保存・この条文アクション・基準日）を持つ。これらは操作の粒度（文書全体に対する操作か、選択中の条に対する操作か）が混在したまま横並びで並び、視線の優先順位が付いていない。結果として、読むための画面であるにもかかわらず操作 UI が本文より目立つ。

一方、右レールの「学習コンテキスト」（メモ／定義語／関連条文／復習カード）は全て「準備中」プレースホルダで、画面の約 1/4 を占有しながら何も提供していない。実際には「この条文を保存／カード／クイズ」といった選択条向けアクションが既にツールバー側に存在しており、機能が無いのではなく導線が二重・未整理の状態である。

本改善は、操作の置き場を「文書レベル＝左レール／選択条レベル＝右レール」という一貫した空間ルールで整理し、中央を読むことに専念させる。

## スコープ

- 中央上部の操作カードを解体し、左右レールへ再配置する（デスクトップ）。
- レール非表示のモバイルでは、操作をボトムシートへ集約する。
- 表示モード切替（読みやすい／原文）をビューワーのローカル状態から永続設定へ移し、設定画面で切り替える。
- 日付表記を `yyyy/mm/dd` に統一する。

### スコープ外

- 右レールの「メモ／定義語／関連条文／復習カード」の中身の実体化（準備中のまま温存）。
- 本文タイポグラフィ・可読性そのものの変更。
- 目次（TOC）の階層表現・現在地追従などの機能拡張。

## 設計

### 1. デスクトップ（`lg` 以上）：2レール分業

中央上部の操作カード（現 `law-viewer-page.tsx` の該当ブロック）を撤去し、中身を意味で振り分ける。

**左レール（文書レベル）** — 上から順に：

- 法令タイトル・法令番号（現状のまま）
- オフライン保存済みバッジ（現状のまま、保存時のみ）
- 条番号ジャンプ（入力＋移動ボタン）
- オフライン保存／保存解除
- 基準日ラベル＋「設定で変更」リンク
- 目次（TOC）

条番号ジャンプは「この文書内で目的の条を探す」導線であり、目次と役割が近いため左レールで隣接させる。

**右レール（選択条レベル）** — 上から順に：

- 「選択中: 第X条」（現状のまま）
- この条文を保存／カードを作る／クイズを生成
- ドリフト表示（`AnchorDriftBadge`。ドリフト・未検出時のみ）
- メモ／定義語／関連条文／復習カード（準備中のまま温存）

条が未選択（`activeArticleNumber === undefined`）のときは、選択条向けの操作群を隠し、「条を選ぶと操作が表示されます」の案内文に置換する。

**中央** — 本文に専念する。各条ホバー時のコピー／URL コピー（`ArticleQuickActions`）は条に密着した操作なので現状維持。

左右レールは既存の `sticky top-14` を維持し、本文をスクロールしても操作が視界に残るようにする。

### 2. モバイル（`lg` 未満）：ボトムシート

レールは `lg:` で非表示のため、操作の出し先をボトムシートに集約する。

- 本文直下にコンパクトな sticky サブバーを置く。下部タブバー（`AppShell` の `fixed bottom-0 md:hidden`）とは干渉しない上部配置とする。
- サブバーのボタンは 2 つ：`[目次]` と `[この条文]`。
- 各ボタンで `Sheet`（`side="bottom"`、既存 `src/shared/ui/sheet.tsx`）を開く。
  - **目次シート**：先頭に文書レベル操作（オフライン保存／保存解除・基準日＋設定リンク・条番号ジャンプ）をまとめ、その下に目次（TOC）を置く。現状の折りたたみ目次はこのシートへ移設する。
  - **この条文シート**：この条文を保存／カード／クイズ／ドリフト表示。アクティブ条があるときのみ有効化し、無いときはサブバーの `[この条文]` を無効化する。

シート内の各アクションは成功時（遷移完了・ダイアログ起動など）にシートを閉じる。保存失敗などのときはシートを維持し、該当操作の近傍にエラーと再試行導線を表示する。

### 3. 表示モードの設定化と日付表記統一

#### 表示モードを永続設定へ

`DisplayPreferences`（`src/core/settings/display-preferences.ts`）に `textDisplayMode` を追加する。既存の `fontSize` / `lineSpacing` / `theme` と同一パターンで実装する。

- 値：`"readable" | "original"`、既定 `"readable"`。
- storage key：`surasura:display:text-mode`。
- `isDisplayTextMode` バリデータ、`getDisplayPreferences` での read＋キャッシュ比較、`setDisplayTextMode` セッター、`handleStorage` のキー判定、`index.ts` の再エクスポート、`useDisplayPreferences` フックへの配線を追加する。
- `core/settings` は `core/viewer` に依存しない。設定側は同値の union（`"readable" | "original"`）を独自に定義し、`core/viewer` の `LawTextDisplayMode` とは構造的互換のまま viewer コンポーネントへ渡す。

設定画面（`src/app/settings-page.tsx`）の静的行「既定の表示 → 読みやすい表示」（既存プレースホルダ）を実 `<Select>`（読みやすい表示／原文表示）に置換し、`textDisplayMode` に配線する。

ビューワーはローカルの `useState<LawTextDisplayMode>("readable")` と表示モードトグル UI を撤去し、`useDisplayPreferences().textDisplayMode` を読むだけにする。読み取った値を `LawDocumentView` / `LawTableOfContents` / `LawNodeList` へ従来どおり渡す。

#### 日付表記を `yyyy/mm/dd` に統一

`src/shared/utils/dates.ts` の `formatIsoDateLabel` を、ISO 日付（`YYYY-MM-DD`）から `YYYY/MM/DD` を返すよう変更する。壊れた値を「不明」とする既存の防御は維持する。

この共通関数はビューワーのほか保存一覧（`saved-page`）・学習カード（`study-cards-page`）・`pages.tsx` でも使われているため、アプリ全体の日付表記が `yyyy/mm/dd` に揃う。

ビューワー内でこれまで生の ISO 文字列を直接表示していた箇所（施行日 `revision.effectiveDate`、基準日 `requestedAsOf`、保存日時 `savedAt`）も、この共通関数経由に寄せて統一する。

## コンポーネント境界

- `law-viewer-page.tsx`：レイアウト（3 カラム grid）とレール内容の振り分け、モバイルサブバー＋シートの開閉状態を保持する。操作カードの解体に伴い、左レール内容・右レール内容・モバイルシート内容をそれぞれ小さなプレゼンテーション単位へ切り出し、肥大化を避ける。
- `core/settings/display-preferences.ts`：`textDisplayMode` の永続化と購読を担う純粋なストア。UI を持たない。
- `core/viewer/*`：表示モードを prop で受け取る現行の契約を維持し、設定化の影響を受けない。
- `shared/ui/sheet.tsx`：既存のまま利用（改変しない）。

## エラー・空状態の扱い

- 条番号ジャンプの不一致・保存失敗・コピー失敗の各メッセージは、対応する操作の新しい配置（左レール／中央／モバイルシート）の近傍に表示する。役割から外れた位置に残さない。
- 右レールで条が未選択のときは案内文を表示する（前述）。
- モバイルの `[この条文]` はアクティブ条が無いとき無効化する。

## テスト方針

- 設定ストア：`textDisplayMode` の read／write／バリデーション（不正値のフォールバック）を table testing で検証する。
- 設定画面：Select 変更でストアが更新され、選択値が反映されることを Testing Library で検証する。
- ビューワー：
  - 設定の `textDisplayMode` に応じて本文表示が切り替わること。
  - 条未選択時に右レールが案内文を表示し、選択時に操作群が現れること。
  - モバイルサブバーからシートが開閉し、シート内の操作（ジャンプ・保存など）が機能すること。
- 日付：既存の日付断定テスト（施行日・取得・基準日・保存日時）を `yyyy/mm/dd` へ更新する。
- いずれもユーザーから見える DOM・公開ストア API・変換結果を対象とし、実装詳細の文字列探索に依存しない。

# 条文からのクイズカード自動生成の設計

Last updated: 2026-07-13

関連ドキュメント:

- 条文アンカーは [2026-07-07-revision-and-asof-design.md](2026-07-07-revision-and-asof-design.md) の二重アンカー + 指紋を共用する。

## 1. 決定事項の要約

- 生成は AI を使わない決定的なルールベース処理とし、`core/study/quiz/` の純関数として実装する。粗い候補はプレビュー UI での取捨選択・編集で吸収する（Issue の「完璧さよりも、根拠条文へ戻れることと、ユーザーが破棄・修正できること」に対応）。
- 生成する種別は穴埋め（fill_blank）、定義語（definition）、条文番号当て（article_number）、正誤（true_false）の 4 種とする。法令名当て・要件効果分解・比較は後続の拡張とする。
- 候補（QuizCandidate）と保存済みカード（StudyCard）を型で分離する。候補は ID も指紋も持たない一時オブジェクトで、ユーザーが採用した時点で初めて指紋を計算して `source: "auto"` の StudyCard になる。破棄は「保存しない」ことで実現し、DB に痕跡を残さない。
- 各生成ルールに `ruleId`（例: `fill-blank@1`）を刻む。スケジューラの `fixed-interval@1` と同じバージョニング流儀で、将来 AI 生成器が同じ候補型を吐く拡張を見込む。
- 条文番号当ての誤答選択肢は同一法令に実在する近傍条文から決定的に選ぶ。`StudyCard` に `choices?: string[]` を追加する（IndexedDB のインデックス変更なし = DB バージョンアップ不要の非破壊変更）。
- UI は保存前プレビュー方式。ビューアの条文アクションに「クイズを生成」を追加し、ダイアログで候補を確認・編集・選択してから保存する。

## 3. 生成ルール仕様

生成器への入力は対象の Article ノード、同一法令の全ノード、法令名。出力は `QuizCandidate[]`。テキストは常に `plainText`（原文保持レイヤー）を使い、読みやすさ変換済みテキストは使わない。文の分割単位は項（Paragraph）とし、文は「。」で区切る。

「削除」「削除。」のみの条文（削除条文）は全ルールで候補ゼロとする。

### 3.1 穴埋め `fill-blank@1`

項テキストの各文から次の 2 カテゴリの語句を抽出し、1 語句 = 1 候補とする。

- 数字・期間: 漢数字または算用数字 + 単位（年・箇月・月・週間・日・時間・歳・円・倍・分の N・人・回・条）の連なり。「五年以下」「三箇月」「二十歳」「十分の一」など。原文は漢数字が主だが、新しい法令の算用数字にも対応する。
- 効果語: 定型パターン辞書による完全一致。最長一致を優先する。
  - 「することができない」「することができる」「しなければならない」「してはならない」「みなす」「推定する」「無効とする」「対抗することができない」「取り消すことができる」

question は対象語句を「（　　）」に置換した文、answer は語句そのもの。同じ語句が同一文に複数回現れる場合はすべて置換する（置換漏れがあると答えが問題文に露出する）。

候補数は 1 項あたり最大 4 件（数字・期間を優先し、残りを効果語で埋める）。question と answer の組が重複する候補は除去する。

### 3.2 定義語 `definition@1`

「「X」とは、Y をいう」パターンを検出する。「この法律において」等の前置きの有無は問わない。X はかぎ括弧で囲まれた語、Y は「とは、」から「をいう」までのテキスト。

- question: 「◯◯法第N条において「X」とは何をいうか？」
- answer: 「Y をいう。」

かぎ括弧のない「X とは、Y をいう」形式は誤検出リスクが高いため v1 では対象外とする。

### 3.3 条文番号当て `article-number@1`

- question: 「次の条文は◯◯法の第何条か？」+ 条文本文の冒頭 snippet。snippet は第 1 項の plainText の先頭 120 字とし、超過分は「…」で省略する。条見出し（caption）は番号の手がかりになるため含めない。
- answer: 対象条の表示名。`node.title`（原文の条見出し。「第七百九条」など）があればそれを使い、なければ「第{number}条」で組み立てる。
- choices: 正解 + 誤答 3 つ。誤答は同一法令の Article ノードを文書順に並べた列から、対象条の直前・直後・5 つ後（末尾側で不足する場合は前方から補う）を決定的に選ぶ。選択肢は文書順に整列する（正解の位置が条によって自然にばらける）。同一法令に 4 条未満しかない場合は存在する分だけの選択肢にする。誤答が 1 つも作れない場合（1 条のみの法令）はこのルールの候補を生成しない。

条文番号当ては 1 条につき 1 候補。

### 3.4 正誤 `true-false@1`

効果語の交換ペア辞書で文中の効果語を 1 箇所だけ置換した誤り文を作る。

| 原文               | 置換後             |
| ------------------ | ------------------ |
| みなす             | 推定する           |
| 推定する           | みなす             |
| することができる   | しなければならない |
| しなければならない | することができる   |

- ×問題: question = 「次の記述は正しいか。」+ 置換後の文、answer = 「×（正しくは「みなす」）」の形式。1 文につき最初の 1 箇所のみ置換し、1 項あたり最大 2 件とする。
- ○問題: 効果語を含む最初の文をそのまま question にし、answer = 「○」。×問題ばかりだと「この形式は常に×」という形式暗記が成立してしまうため、1 条につき 1 件だけ混ぜる。

## 4. 型とモジュール構成（core/study/quiz/）

```
src/core/study/quiz/
├── types.ts              # QuizCandidate、GenerationContext
├── generate.ts           # generateQuizCandidates() オーケストレーター
└── rules/
    ├── fill-blank.ts
    ├── definition.ts
    ├── article-number.ts
    └── true-false.ts
```

```ts
// 保存前の一時オブジェクト。ID・指紋・タイムスタンプを持たない
interface QuizCandidate {
  type: StudyCardType;
  ruleId: string; // "fill-blank@1" など。生成器のバージョン識別子
  question: string;
  answer: string;
  choices?: string[]; // article_number のみ
  sourceNodeId: string; // 生成元 Article ノードの id
}

interface GenerationContext {
  lawTitle: string;
  nodes: readonly LawNode[]; // 同一法令の全ノード。項テキスト参照と近傍条文の選択に使う
}

const generateQuizCandidates = (article: LawNode, context: GenerationContext): QuizCandidate[];
```

- `LawNode.children` は id 配列（正規化ストア形式）のため、generate.ts 内で id → ノードの Map を構築して項テキストと近傍条文を解決する。
- 出力順は定義語 → 穴埋め → 条文番号当て → 正誤。プレビューで種別ごとにグループ表示する前提の並びとする。
- `core/study` は既存どおり `core/domain` のみに依存する。新しい依存は増えない。

## 5. StudyCard の変更と保存経路

- `StudyCard` に `choices?: string[]` を追加する。多択の選択肢を question 文字列へ焼き込まず構造化して持つことで、#29 の復習画面が選択肢をボタンとして描画できる。
- IndexedDB はオブジェクトをそのまま保存し、choices にインデックスは張らないため、スキーマ変更・DB バージョンアップ・移行は不要。既存カードは choices が undefined のままでよい。
- `SavedDataExport` はカードをそのまま含むため、任意フィールドの追加は後方互換であり version は上げない。
- 採用時の変換: 既存の作成ダイアログと同じアンカー構築パターンに従う。`computeArticleFingerprint(node.plainText)` → `{ lawId, article, revisionId, fingerprint }` → `putStudyCard`。`source: "auto"`、`tags: []`、`examPinned: false`、explanation は付けない。

## 6. UI 設計

### 6.1 ビューア導線

法令ビューアの条文アクション（「カードを作る」の並び）に「クイズを生成」を追加する。`LawNodeList` の `renderArticleActions` 拡張点をそのまま使い、`law-viewer-page.tsx` が持つ `state.nodes` と対象ノードをダイアログへ渡す。

### 6.2 クイズ生成ダイアログ（QuizGenerateDialog）

props は StudyCardCreateDialog と同型（lawId / lawTitle / revisionId / articleNumber / node / storageRepository）に `nodes` を加えたもの。

- 開いたときに `generateQuizCandidates` を実行し、候補を種別ごとにグループ表示する（見出しは `studyCardTypeLabels` を共用）。生成は同期・軽量なのでローディング状態は設けない。
- 各候補はチェックボックス付きで既定 ON。「すべて解除」「すべて選択」のトグルを置く。
- question / answer は候補ごとに textarea でインライン編集できる。choices は読み取り専用のリスト表示とする（選択肢の編集は保存後の詳細ページ改修も含めて対象外）。
- フッターに「選択した N 件を保存」ボタン。選択 0 件のときは無効化する。
- 保存時は指紋を 1 回計算し、選択された候補を逐次 `putStudyCard` する。途中で失敗した場合は「N 件保存した時点で失敗した」ことをダイアログ内に表示し、ダイアログは閉じない（保存済み分は残る）。
- 選択中の候補に trim 後空の question / answer があれば保存せずエラー表示する。
- 候補ゼロ（削除条文、抽出パターンに一致しない条文）のときは「候補が見つかりませんでした」と手動作成（「カードを作る」）への案内を表示する。
- 閉じるときは生成結果と編集内容を破棄する（既存ダイアログの「途中入力を破棄」と同じ方針）。

### 6.3 カード詳細ページの小改修

`/study/cards/$cardId` で choices を持つカードは選択肢を読み取り専用のリストとして表示する。編集フォームの対象には含めない。

## 7. エラー処理

| 局面                         | 方針                                                                               |
| ---------------------------- | ---------------------------------------------------------------------------------- |
| 生成処理                     | 純関数であり例外を投げない。パターン不一致は候補ゼロとして正常系で扱う             |
| 保存失敗（IndexedDB）        | ダイアログ内にエラーと保存済み件数を表示し、ダイアログは閉じず選択・編集を保持する |
| 編集後の空 question / answer | 保存せずダイアログ内にバリデーションエラーを表示する                               |
| 誤答選択肢が 3 つ作れない    | 作れる分だけの選択肢にする。1 つも作れなければ候補自体を生成しない                 |

## 8. テスト戦略

- 生成ルール（table testing）: LawNode の fixture を使い、代表ケースと境界ケースを表形式で検証する。
  - 穴埋め: 数字・期間の抽出（漢数字・算用数字・「分の N」）、効果語の最長一致（「することができない」が「することができる」に誤マッチしない）、同一語句の複数出現の全置換、1 項 4 件の上限、重複除去。
  - 定義語: 前置きあり・なしの検出、かぎ括弧なし形式を対象外とすること。
  - 条文番号当て: snippet の 120 字省略、caption 非含有、近傍選択（先頭条・末尾条・枝番「242の2」）、選択肢の文書順整列、条数不足時の縮退、1 条のみの法令で候補なし。
  - 正誤: 交換ペアの置換、1 文 1 箇所の制限、○問題が 1 条 1 件であること。
  - 共通: 削除条文で全ルール候補ゼロ。
- ダイアログ（Testing Library）: 候補のグループ表示、チェック解除で保存対象から外れること、インライン編集が保存内容に反映されること、保存で `putStudyCard` に渡る内容（`source: "auto"`、二重アンカー + 指紋、choices）、空 question のバリデーション、候補ゼロの空状態、キャンセルで何も保存されないこと。
- 実画面確認: `playwright-cli open --headed` で条文から生成 → 取捨選択 → 保存 → 一覧・詳細で確認する一連フローを確認し、スクリーンショットを PR に添付する。

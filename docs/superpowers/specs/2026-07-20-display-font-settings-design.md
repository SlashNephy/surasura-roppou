# 表示フォント設定の設計

## 1. 目的

現行のフォントは、`--font-sans` と `--font-serif` に固定したシステムフォントスタックであり、利用者は変更できない。

Issue #134 では、条文を読む領域の**本文フォント**と、アプリ全体の**UI フォント**を設定画面から変更できるようにする。
二つを独立した設定項目とし、条文だけを別のフォントで読む使い方と、アプリ全体の書体を変える使い方の両方に応える。

## 2. 完了境界

次の項目を実装対象とする。

- 本文フォントと UI フォントの二項目の設定
- システム標準と Google Fonts 由来の五書体からなる共通の選択肢
- Fontsource パッケージによるフォントの自己ホスト
- 名前空間付き `localStorage` への保存と次回起動時の復元
- 初回描画前の DOM 属性反映によるちらつき防止
- service worker によるフォントファイルのオンデマンドキャッシュ
- 設定画面での選択中フォントのプレビュー表示

次の項目は実装対象に含めない。

- Google Fonts CDN など外部オリジンからのフォント読み込み
- 利用者による任意フォント名の自由入力
- 見出し用フォント（UI の `font-serif`）の独立した設定
- 字間、字送り、ウェイトの調整項目
- フォント設定のサーバー同期

見出しの `font-serif` は現行デザインを維持し、本文フォント設定に追従させない。
見出しまで変えたい要望が出た場合は、別 Issue で扱う。

ウェイトの調整は対象外だが、将来の追加候補として想定しておく。
Noto の二書体を variable フォントで同梱するため、本文の太さ設定を後から足す場合も追加のフォント取得は発生せず、同じ DOM 属性方式に一項目を加えるだけで実現できる。

## 3. 選択肢と既定値

本文フォントと UI フォントは、同じ選択肢の集合から選ぶ。
明朝で UI を使う選択も、ゴシックで条文を読む選択も妨げない。

| 内部値           | 表示名           | 分類         | 提供元                               |
| ---------------- | ---------------- | ------------ | ------------------------------------ |
| `system`         | システム標準     | —            | 現行のシステムフォントスタック       |
| `noto-serif-jp`  | Noto 明朝        | 明朝         | `@fontsource-variable/noto-serif-jp` |
| `biz-udmincho`   | BIZ UD 明朝      | UD 明朝      | `@fontsource/biz-udmincho`           |
| `zen-old-mincho` | ZEN オールド明朝 | オールド明朝 | `@fontsource/zen-old-mincho`         |
| `noto-sans-jp`   | Noto ゴシック    | ゴシック     | `@fontsource-variable/noto-sans-jp`  |
| `biz-udgothic`   | BIZ UD ゴシック  | UD ゴシック  | `@fontsource/biz-udgothic`           |

初回起動時は、両項目とも `system` を使う。
`system` は現行のフォントスタックそのものであり、設定に触れない利用者の表示は変わらない。

Noto の二書体は variable フォントとして同梱し、既存の全ウェイト指定に追従する。
静的フォントは提供されているウェイトだけを同梱し、提供のない中間ウェイトはブラウザの合成表示に任せる。

## 4. 保存契約

フォント設定は、既存の表示設定と同じ契約で保存する。

| 項目         | キー                        |
| ------------ | --------------------------- |
| 本文フォント | `surasura:display:law-font` |
| UI フォント  | `surasura:display:ui-font`  |

初回起動時はキーを作らず、読み取り時に既定値を補う。
保存値は既知の union に含まれる文字列だけを有効とし、不正な値を読んだ場合はその項目だけを既定値へ戻す。
同一タブの変更は購読者へ直ちに通知し、別タブの変更は `storage` イベントで受け取る。

## 5. フォント資産の同梱と読み込み

五書体の Fontsource パッケージを依存へ追加し、それぞれの CSS を静的に import する。
npm パッケージとして管理するため、Renovate が更新を追跡できる。

`@font-face` は宣言しただけでは取得が発生しない。
CSS には `unicode-range` で分割されたサブセットの宣言が並び、選択中のフォントが実際の描画で使われたときに、表示中の文字を含むチャンクだけがダウンロードされる。
このため、全書体の CSS を最初から読み込んでも、通信が発生するのは選択中の書体の使用分に限られる。

未取得のチャンクは `font-display: swap` によりフォールバックフォントで表示し、文字が読めない時間を作らない。

このコストは、CSS バンドルの `@font-face` 宣言分の増加と、ビルド成果物へのフォントファイル追加である。
配信量は選択中フォントの使用チャンク分にとどまるため、フォントを変更しない利用者の通信量は増えない。

## 6. CSS の反映

`@theme inline` に本文フォント用の `--font-law` を追加し、既定値を `var(--font-serif)` とする。
条文を読む領域のコンポーネントは、`font-serif` の代わりに `font-law` ユーティリティを使う。

置き換えの対象は、条文本文、条文の目次、条文カードの本文表示である。
ページ見出しやアプリシェルの `font-serif` は UI のデザイン要素として維持する。
実装時に `font-serif` の使用箇所を洗い出し、条文表示に当たるものだけを `font-law` へ移す。

フォントの切り替えは、既存の文字サイズと同じ `document.documentElement` の属性方式で行う。

- `data-law-font="system | noto-serif-jp | ..."` が `--font-law` を上書きする
- `data-ui-font="system | noto-sans-jp | ..."` が `--font-sans` を上書きする

`system` のとき、および属性がないときは、現行のフォントスタックのまま表示する。

## 7. React への接続

既存の表示設定 Provider に本文フォントと UI フォントを追加し、layout effect で DOM 属性へ反映する。
設定の読み取りと更新は既存の app 専用 hook を通し、設定画面が `localStorage` や DOM 属性を直接操作しない構造を保つ。

`index.html` の inline script は、現在テーマのちらつき防止を担っている。
このスクリプトを拡張し、初回描画前に `data-law-font` と `data-ui-font` も反映する。
`localStorage` へのアクセス失敗時は属性を付けず、既定のフォントで表示する。

## 8. PWA とオフライン

フォントファイルは service worker のプリキャッシュへ含めない。
全書体の全サブセットをプリキャッシュすると起動時のキャッシュ容量を圧迫するためで、OCR 資産を `globIgnores` で除外している既存方針と同じ判断である。

代わりに Workbox の `runtimeCaching` を追加し、同一オリジンのフォントファイル要求を CacheFirst で扱う。
`maxEntries` を設定し、キャッシュが際限なく増えない形にする。

オンラインで一度表示した文字のチャンクはキャッシュから配信される。
オフラインで未キャッシュのチャンクに当たった場合は、`font-display: swap` によりシステムフォントで表示し、内容が読めない状態を作らない。

## 9. 設定画面

設定画面の「表示」グループに、本文フォントと UI フォントの native select を既存の行レイアウトで追加する。
選択は保存ボタンを介さず、変更時に直ちに保存して全画面へ反映する。

本文フォントの行には、選択中のフォントで固定のプレビュー文を一行表示する。
プレビュー文には日本国憲法前文の冒頭を使い、選んだ書体で条文がどう見えるかをその場で確認できるようにする。

## 10. データ境界

フォント設定は表示レイヤーだけに作用する。

法令ドメインモデル、正規化済みノード、IndexedDB の保存値にはフォント設定を渡さない。
コピーと export の内容はフォント設定に依存しない。
フォント設定を URL、ログ、例外メッセージに含めない。

## 11. テスト方針

設定ストアは既存の表示設定と同じ形で次を検証する。

- 保存値がない場合の既定値
- 各選択肢の保存と復元
- 不正値を項目単位で既定値へ戻す処理
- 同一タブの購読通知
- 別タブの `storage` イベント

Provider のテストでは、設定値と `document.documentElement` の `data-law-font`、`data-ui-font` 属性を検証する。

設定画面の Testing Library テストでは、実際に select を変更して次を確認する。

- 選択値の可視表示
- `localStorage` への保存
- DOM 属性の即時変更
- プレビュー文の表示

ソースコードや設定ファイルの文字列を探索するだけのテストは追加しない。

## 12. 実画面検証

実装後は、システムにインストールされた `playwright-cli` で実画面を確認する。
dev server は依存の再最適化でリロードが入るため、preview build を使う。

次の操作を確認する。

1. 本文フォントを各書体へ順に切り替え、法令本文と目次だけが変わることを確認する。
2. UI フォントを切り替え、アプリ全体の地の文が変わることを確認する。
3. 設定画面のプレビュー文が選択に追従することを確認する。
4. reload 後に選択値が復元され、初回描画からちらつきなく適用されることを確認する。
5. ネットワークパネルで、選択中フォントのチャンクだけが取得されることを確認する。

法令ビューアの確認には、e-Gov API から取得した実レスポンスを使う。
desktop と mobile のスクリーンショットを保存し、`gh image upload` で PR 本文へ添付する。

## 13. 完了前の確認

コミット直前にプロジェクト標準の品質チェックを実行する。

```bash
pnpm run typecheck
pnpm run lint
pnpm run format:check
pnpm test
git diff --check
```

PR 作成前に Antigravity review を一回実行する。

```bash
pnpm run review:antigravity
```

Antigravity review を踏まえて修正した場合は、通常の品質チェックを再実行し、Antigravity review は再実行しない。

# 品質基準監査 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Issue #40 で要求されたアクセシビリティ、性能、OCR lazy load、プライバシー、ローカルデータ削除の最低基準を一つの運用文書にまとめ、現行実装を実レスポンスと実ブラウザで監査し、未達事項を追跡可能な後続 Issue にする。

**Architecture:** 製品コードには変更を加えず、`docs/quality-baseline.md` を規範、監査結果、実装者チェックリストの単一の入口にする。現在の実装は production build、e-Gov API の実レスポンス、headed Chrome の実操作、ブラウザストレージとネットワークの観測で判定する。表示設定と一括削除は必須の後続 Issue、その他の未達は監査結果に応じた条件付き Issue とし、`docs/design-doc.md` から基準文書へ導線を張る。

**Tech Stack:** Markdown、React 19、Vite 8、TypeScript 6、pnpm、Vitest、ESLint、Prettier、Node.js 24、system-installed `playwright-cli`、e-Gov 法令 API v2、GitHub CLI / GitHub connector

---

## 実行原則

- 各実装 Task は新しい実装 Subagent に渡し、その後に別の仕様適合レビュー Subagent、さらに別のコード品質レビュー Subagentを順番に割り当てる。
- Subagent にはこの計画、設計仕様書、Issue #40、`AGENTS.md` の絶対遵守事項を明示する。
- レビュー指摘は客観的に検証し、妥当な指摘だけを同じ実装 Subagent に戻して修正させる。修正後は同じ段階のレビューをやり直す。
- 一つの Task の変更、検証、コミット、レビューを完了してから次の Task に進む。
- GitHub Issue 作成、PR 作成、スクリーンショット upload、サスペンド予約のような外部状態の変更は root agent が行う。
- 既存のユーザー変更を戻さない。計画外の製品コード変更が必要になった場合は作業を止め、ユーザーへ確認する。

- [ ] **Step 2: アクセシビリティ基準を書く**

次を規範語「必須」「禁止」「推奨」で明確にする。

- 目標は WCAG 2.2 Level AA。
- マウスで到達できる全操作はキーボードでも到達できる。
- `Tab` / `Shift+Tab` は論理順、`Enter` / `Space` は操作、`Escape` は一時 UI の閉鎖、矢印キーは該当 widget の WAI-ARIA pattern に従う。
- フォーカスを視認でき、モーダルを閉じた後は起点へ戻す。
- semantic landmark、見出し階層、accessible name、状態通知を維持する。
- 200% 拡大と幅 320 CSS px 相当で情報や操作を失わない。
- タッチ対象は原則 44 x 44 CSS px 以上とする。例外は本文中の inline link など WCAG の例外条件を満たすものだけにする。
- 自動監査だけで適合判定せず、実キーボード操作とデスクトップ / モバイル表示を確認する。

公的基準へのリンクとして WCAG 2.2、Understanding Keyboard、Focus Visible、Reflow、Target Size (Minimum) を一次資料へ直接リンクする。

- [ ] **Step 3: 表示設定の方針を書く**

次を明記する。

- 文字サイズは少なくとも標準、大きめ、特大を提供する。
- 行間は少なくとも標準、広めを提供する。
- テーマはライト、ダーク、OS 設定を提供する。
- 設定は端末ローカルに保存し、初回は OS 設定を尊重し、本文、目次、検索結果、ダイアログで一貫して反映する。

- [ ] **Step 4: 性能基準を書く**

次を明記する。

- 製品目標は Core Web Vitals の LCP 2.5 秒以下、INP 200 ms 以下、CLS 0.1 以下の good threshold。
- Issue #40 のローカル監査は Civil Code `lawId=129AC0000000089` を代表的な大規模法令に使う。
- production build と warm browser session を使用する。
- API 応答完了から最初の `article` 表示までを 3 回測り、中央値が 1,000 ms を超えたら未達とする。
- 条文本文の表示切替や目次操作を 3 回測り、中央値が 200 ms を超えたら未達とする。
- 法令本文の変換または描画に起因する 50 ms 超の long task、クラッシュ、コンソール error が 1 件でもあれば未達とする。
- 実験室測定は実利用者の Core Web Vitals を保証しないため、将来 field data を取得する場合はプライバシー基準を先に満たす。

- [ ] **Step 5: OCR lazy load 基準を書く**

次の通信境界を明記する。

- ホーム、法令閲覧、検索、復習、設定を使うだけでは OCR worker、WASM、言語モデル、OCR 実行専用 chunk を取得しない。
- `/scanner` を開いただけ、画像を選んだだけでは worker、WASM、言語モデルを取得しない。
- ユーザー同意後に OCR 実行を開始した時点で初めて OCR runtime とモデルを取得する。
- module dynamic import だけでなく Service Worker precache も監査対象にする。
- OCR runtime / model のリクエストを URL と initiator で記録する。
- 既存 cache が結果を隠さないよう、初回取得の監査では Cache Storage と Service Worker 状態を記録する。

- [ ] **Step 6: プライバシー、セキュリティ、削除基準を書く**

データ境界を次の列で表にする。

```markdown
| データ | 端末保存 | 外部送信 | analytics | 削除対象 | 備考 |
```

最低限、閲覧履歴、ブックマーク / 保存リスト、メモ、復習履歴、OCR 画像、OCR テキスト、表示設定、法令 cache、検索 index を列挙する。次を明記する。

- OCR 画像は既定で端末内処理し、永続保存も外部送信もしない。
- object URL と worker は置換、破棄、キャンセル、画面離脱に応じて解放する。
- OCR テキストはユーザーが確認できるセッション情報として IndexedDB に保存し得るが analytics へ送らない。
- 閲覧履歴、メモ、OCR テキストを analytics に送ることを禁止する。
- analytics を新設する場合は opt-in、データ分類、目的、保持期間、削除方法、送信先、同意撤回を別 Issue で合意する。
- 外部取得データは text node として扱い、HTML 注入を避ける。
- 「ローカルデータをすべて削除」は主 IndexedDB、Cache Storage、Service Worker、localStorage、sessionStorage、表示設定、OCR セッションを対象にし、実行前に復元可能データと不可逆データを説明する。
- 現行 export は OCR セッションと表示設定を復元できないため、一括削除 UI ではその差を明示する。

- [ ] **Step 3: e-Gov Civil Code の実レスポンスを 3 回取得する**

同じコマンドを 3 回実行する。

```bash

import { normalizeEgovLawText } from "./src/core/egov/lawText.ts";

const url = "https://laws.e-gov.go.jp/api/2/law_data/129AC0000000089?law_full_text_format=json&response_format=json";
const fetchStartedAt = performance.now();
const response = await fetch(url);
const body = await response.text();
const fetchFinishedAt = performance.now();
if (!response.ok) {
  throw new Error(`e-Gov request failed: ${response.status}`);
}
const payload = JSON.parse(body);
const normalizeStartedAt = performance.now();
const nodes = normalizeEgovLawText(
  payload.law_full_text,
  payload.law_info.law_id,
  payload.revision_info.law_revision_id,
);
const normalizeFinishedAt = performance.now();
console.log(JSON.stringify({
  url,
  status: response.status,
  bytes: new TextEncoder().encode(body).byteLength,
  fetchMs: Number((fetchFinishedAt - fetchStartedAt).toFixed(2)),
  revisionId: payload.revision_info.law_revision_id,
  lawTitle: payload.law_full_text.law_title,
  nodeCount: nodes.length,
  normalizeMs: Number((normalizeFinishedAt - normalizeStartedAt).toFixed(2)),
}, null, 2));
NODE
```

- [ ] **Step 2: サスペンドを遅延予約する**

最終報告を Codex から送信できる短い猶予を設けて、systemd transient timer で `systemctl suspend` を一度だけ予約する。予約前に他のユーザー session や inhibitor を確認し、予約コマンドが成功したことを証跡として保存する。予約に失敗した場合は、成功したと報告せず理由を明記する。

実行候補:

```bash
systemd-inhibit --list
systemd-run --user --unit=codex-issue40-suspend --on-active=45s /usr/bin/systemctl suspend
systemctl --user status codex-issue40-suspend.timer --no-pager
```

環境が user transient timer から suspend を許可しない場合だけ、system scope の transient timer を使う。即時 suspend は最終報告の送信を妨げるため行わない。

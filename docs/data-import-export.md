# データのエクスポートとインポート

すらすら六法は、端末に保存した7分類のデータを一つのJSONファイルにまとめ、別の端末へ移動できます。

## エクスポートとインポートの手順

設定の「エクスポート / インポート」を開きます。

エクスポートでは、「JSONをエクスポート」を選ぶと現在の保存データをファイルに書き出します。
保存リストにある同名のボタンは、このエクスポートを直接実行するショートカットです。

インポートでは、`version: 2`のJSONファイルを選択します。
ファイルの`version`、出力日時、7分類ごとの件数がプレビューに表示されるため、内容を確認してから「この内容をインポート」を選びます。
ファイルを選択しただけでは保存データを変更しません。

## 移動するデータ

一つのJSONファイルに次の7分類を含めます。

- 保存法令本文
- ブックマーク
- コレクション
- メモ
- 学習カード
- 回答ログ
- 学習セッション

`CardSchedule`（復習予定の派生データ）はファイルに含めず、インポート後に`ReviewLog`（回答ログ）から再構築します。

## プライバシー

JSONの作成、検証、インポートは端末内で処理し、サーバーへ送信しません。
保存法令本文を1件でも読み出せない場合は、不完全なファイルを成功扱いせず、エクスポート全体を中止します。

ただし、書き出したファイル自体には利用者が保存した法令本文、メモ、学習履歴などが入ります。
ファイルの保管場所と共有先に注意してください。

## 対応バージョンと正本スキーマ

現在対応する形式は`version: 2`だけです。
`version: 1`は利用者向けの配布実績がないため、変換も受理もしません。

`version: 2`の正本スキーマは、[saved-data-export-v2.schema.json](./schemas/saved-data-export-v2.schema.json)です。
連携実装は、この正本スキーマに対してファイル全体を検証します。

任意フィールドの追加を含め、JSON構造を変更するときは`version`を上げます。
新しい`version`への対応は明示的に実装し、未知の`version`を推測して処理しません。

## スキーマ外の受理条件

インポートでは、正本スキーマによる検証の後に、次の意味検証を行います。

- 各分類で使うIDは、同じファイル内の同じ分類で重複できません。
- 保存法令本文の`law.lawId`と`revision.revisionId`はそれぞれ重複できず、`node.id`は全保存法令本文を通して一意である必要があります。
- 各`ReviewLog.cardId`は、同じファイル内にある`StudyCard.id`を参照する必要があります。
- 保存法令本文ごとに、`law.lawId`は`revision.lawId`および各`node.lawId`と一致し、`revision.revisionId`は各`node.revisionId`と一致する必要があります。
- 保存法令本文の各ノードは、同じ本文内に実在する親子を相互に参照し、子の重複や循環を含まない木構造である必要があります。

一方、`Collection.bookmarkIds`、`ReviewLog.sessionId`、`StudySession.cardIds`が同じファイル内に対応先を持たない場合も受理し、その参照値を保持します。

このため、正本スキーマへの適合だけではインポートの受理を保証しません。

## マージと失敗時の動作

マージでは、同じ分類内で識別子が一致するデータをファイル側の内容で上書きします。
ファイルに含まれない既存データは、そのまま保持します。

ファイル全体の正本スキーマ検証と意味検証を通過したデータは、一つのトランザクションで書き込みます。
途中で失敗した場合はトランザクション全体を取り消し、保存データを一件も変更しません。
